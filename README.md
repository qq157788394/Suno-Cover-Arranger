## 解决的痛点

我们深知 AI 音乐创作者面临的两大核心难题：**“编曲描述难”** 和 **“歌词没灵魂”**。

### 🎸 编曲与翻唱痛点

**痛点一：Suno 的 Styles 字段写不好**
做过几次 Suno 翻唱的人，基本都会遇到：

- 知道自己想要“张惠妹那种主歌隐忍、副歌爆炸”，但不会用英文系统地写出来
- 写出来的 Styles 要么太短、信息太少，要么东一块西一块，没有结构
- 不知道哪些词是 Suno 比较“听得懂”的编曲/动态词汇
  _结果：模型发挥空间过大，生成极不稳定。_

**痛点二：Lyrics 字段的“高级用法”很难落地**
Suno 官方推荐使用属性注释（如 `[Vocal: ...]`），但普通人很难上手：

- 不知道应该有哪些 Key（Vocal / Dynamics / Texture / Mood...）
- 不会用英文把“这一段主歌更克制”这种感觉，拆成结构化的 `[Key: Value]`
- 注释与 Styles 描述矛盾，导致风格分裂
  _结果：大部分人退回到只放纯歌词，浪费了 Suno 的潜力。_

**痛点三：歌词不敢随便交给 AI 改写**
很多 AI 工具默认会“润色”歌词，这对翻唱是灾难：

- 翻唱需要保留原汁原味，不能乱改词
- 创作者只想要 Styles 和属性注释，不想要改动歌词主体

**痛点四：段落结构标签混乱**
真实使用时，标签五花八门（【主歌】、[verse]、[Chorus]...），而 Suno 官方推荐标准英文标签（[Verse 1], [Chorus], [Bridge]）。手工统一标签既费时又枯燥。

### ✍️ 原创歌词痛点（新！）

**痛点五：AI 写词“味同嚼蜡”**
市面上的 AI 写词工具充满了“陈词滥调”：

- 动不动就是“岁月的长河”、“霓虹灯闪烁”、“心中的涟漪”
- 辞藻华丽但逻辑空洞，完全没有画面感
- 缺乏个人风格，写谁都像是在写“AI 朗诵”

**痛点六：不懂方言与韵律**

- 写粤语歌时，混入“的、是、什么”等普通话词汇，甚至出现倒字（不协音）
- 为了强行押韵，写出逻辑不通的句子（凑韵）
- 长短句没有呼吸感，歌手根本唱不进去

## 核心功能

### 🎵 模块一：大师做编曲

1.  **智能生成 Styles**：根据中文风格描述，自动生成符合 Suno 官方术语体系的高质量英文 Styles。
2.  **结构化 Lyrics 属性**：自动识别并标准化歌词段落标签，为每段生成结构化的属性注释。
3.  **严格保护歌词原文**：只在歌词前添加工程标注，不修改、不润色歌词主体。

### 📝 模块二：大师写歌词（New!）

1.  **11 位大师灵魂注入**：集结 **周杰伦、毛不易、Beyond(黄家驹)、林夕、方文山、罗大佑、李宗盛、唐恬、陈曦、飞儿乐团、刘卓辉** 的创作模型。
2.  **LLM 深度自检机制**：后台自动扫描并修正“AI 味”、押韵错误及方言问题，拒绝洗稿。
3.  **方言协音支持**：支持生成地道的 **粤语 (Cantonese)** 和 **闽南语 (Hokkien)** 歌词。
4.  **参考歌曲投喂**：支持输入目标参考歌曲（Reference Song），让 AI 精准锚定情绪与结构。

### 🛡️ 架构优势

- **纯前端架构**：无后端、不存库、不打日志，完全保护用户隐私。

## 功能使用指南

### 1. 翻唱编曲模式

1.  选择 **[大师做编曲]** 标签页。
2.  输入目标翻唱歌手、风格备注。
3.  粘贴原曲歌词。
4.  点击"生成提示词"，复制 Styles 和 Lyrics 到 Suno。

### 2. 大师写歌词模式（新功能）

1.  选择 **[大师写歌词]** 标签页。
2.  **选择大师**：从 11 位大师中选择一位（如：周杰伦）。
3.  **设置参数**：
    - **贴近度**：调节模仿力度（1-5 档）。
    - **参考歌曲**：输入你想模仿神韵的歌曲名（如《蜗牛》）。
4.  **输入素材**：提供故事背景或创作主题。
5.  点击生成，等待大师“思考”并自检，输出最终歌词。

### 3. 记录管理

- **查看/删除**：点击左侧“生成记录”，支持按语言、歌手、模式筛选或删除历史记录。

## 分支管理规范

### 1. 分支命名与作用

- **develop 分支**：开发分支，用于开发新功能、修复 bug
- **main 分支**：主分支，用于存储稳定版本，只接收来自 develop 分支的合并
- **gh-pages 分支**：用于部署 GitHub Pages，由 CI/CD 或手动部署命令自动生成

### 2. 开发流程

1. 在 develop 分支上开发新功能
2. 测试通过后，合并到 main 分支
3. 使用 `pnpm deploy` 命令部署到 gh-pages 分支
4. 确保 main 分支与 develop 分支保持同步

### 3. 部署规范

- **部署命令**：`pnpm deploy`
- **部署前检查**：
  - 确保代码通过 lint 检查：`pnpm lint`
  - 确保构建成功：`pnpm build`
  - 确保功能测试通过
- **部署后验证**：
  - 访问 GitHub Pages 地址验证页面显示正常
  - 测试核心功能是否正常

## 常见问题解决

### 1. GitHub Pages 页面不显示

- 确保使用了 `pnpm deploy` 命令部署
- 检查 gh-pages 分支是否存在
- 检查 GitHub Pages 设置是否正确
- 等待几分钟，GitHub Pages 可能需要时间更新

### 2. 生成的记录不显示

- 确保浏览器支持 IndexedDB
- 检查浏览器开发者工具的控制台，查看是否有错误
- 清除浏览器缓存，刷新页面

### 3. 筛选功能不生效

- 确保输入了正确的筛选条件
- 检查浏览器开发者工具的控制台，查看是否有错误
- 刷新页面后重试
